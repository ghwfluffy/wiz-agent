# Phase 08: Approvals And Notification Policy

## Goal

Give the agent a safe way to ask for permission, clarification, or notification
delivery without becoming noisy. Approvals should work from both web UI and
owner messaging channels.

## Approval Types

Start with:

- send outbound message
- update task schedule
- trust sender
- block sender
- write important memory
- cross-app write action
- delete/archive data

Each approval should include:

- source run id
- source message/task/memory path
- action type
- proposed payload
- risk level
- human-readable summary
- expiration time
- status
- decided by
- decided at

## Notification Policy Memory

Create `/assistant/notification-policy.md` with owner-editable rules:

- quiet hours
- urgent categories
- daily briefing preference
- newsletter synthesis preference
- max messages per day
- preferred channel order
- when to batch questions
- when to ask immediately

The agent may propose changes, but host code applies them only through memory
write tools and appropriate approval rules.

## SMS/Email Commands

Support simple replies:

- `YES`
- `NO`
- `EDIT <text>`
- `LATER`
- `DETAILS`
- `TRUST`
- `BLOCK`

Host code should parse commands against the most recent pending approval for
that owner channel. Do not let arbitrary inbound messages select approval IDs
unless they include a signed short code generated by host code.

## Web Approval Inbox

Add UI:

- list pending approvals
- group by risk/source
- show exact proposed action
- approve/reject/edit
- inspect linked run/message/task/memory
- bulk reject stale approvals

## Agent Integration

Tools should be able to return:

- executed immediately
- queued approval
- rejected by policy
- needs clarification

For high-risk actions, the tool should create an approval instead of executing.

## Tests

Backend:

- approval creation for high-risk outbound.
- approval required for cross-app write action.
- SMS `YES` approves only current owner approval.
- expired approval cannot execute.
- rejected approval records audit.
- edit creates updated payload and audit.

Frontend:

- approval inbox renders pending actions.
- approve/reject calls API.
- edited approval shows changed payload.

## Completion Criteria

- High-risk actions are approval-gated.
- Owner can approve from web or owner messaging channel.
- Notification policy is memory-backed and visible.
- `./scripts/validate.sh` passes.
